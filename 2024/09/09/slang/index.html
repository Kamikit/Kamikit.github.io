2.3 GPU架构特性：

  • SIMD架构：避免分歧控制流(if)，因此类似 dynamic dispatching 和虚函数不被支持
  • 多线程：切换线程的上下文尽可能轻量化
  • 高度流水线化：尽可能减少状态的改变，避免因为状态改变而产生的流水线 stall 以及 CPU传递硬件指令而产生的额外开销

2.5 Shader System设计需求：

  • 非耦合模块化的方式实现 shading feature
  • 代码基础要能够便于扩展，添加新的 shading feature (例如新类型的光照或材质)
  • Shading system 要能够生成并使用特化的 shader 变体，用来进行更高效的GPU运算
  • Shading system 要最小化GPU状态改变的数量，使用 parameter block 来进行更高效的参数传递

3.1 Shader Component 设计思路：

  • 一种 shading feature 对应一个 component type，封装其所需的参数和 shader 代码
  • component type 用于特化该 shader 代码
  • 参数通过独特的 parameter block 进行传递

3.1.1 着色器代码与参数封装
  • component type: 封装该 shading feature 的参数及 shader 代码(例如相机、某种材质或光照)
  • component interface: 声明 component 的基础方法(例如光照、材质)
3.1.2 通过 component type 创建特化的着色器代码

    • entry point并不是完整的，其中包含了许多槽位，可以用 component type (例如 Camera) 或 component interface (例如 ILighting、IMaterial) 来进行表示

3.2 着色器组件实现：

  • slang 将着色系统设计模式(shader component)的实现与着色器语言和编译器运行时分离开，着色器编译器负责生成特化的代码，提供着色器代码的反射信息，着色系统开发者负责使用这些编译服务来实现各种符合系统需求的设计模式。这种职责分离能够保证着色系统设计模式能够独立于着色语言和编译器来进行定制和演化，从而允许着色器语言本身能够在不同的着色系统普遍适用。
3.2.1 编译和加载着色器代码
  • 着色系统首先需要加载一个着色器代码库，其中定义了所有的 component type 和 entry point，可以调用对应的方法通过名字在库中查找 component type 或 entry point

  • 组件实例存在的问题是 host language 的编译器无法为组件实例提供任何类型检查。

3.2.3 绘制物体
  • 每当向 GPU 流水线提交绘制指令，着色系统都需要通过某些绑定函数来选择 entry point 和 component instances，这些绑定函数是由着色系统定义的，并非实际的 GPU 指令。

  • 为了避免反复编译着色器变体，着色系统可以在着色器程序 cache 中进行查找，看是否已经存在该 entry point 和 component instances 的组合，如果没有，则执行编译，生成对应的着色器变体并存入 cache，随后着色系统就可以通过设置 GPU 流水线状态来使用该着色器程序。

  • 着色系统可以选择将所有可能使用的着色器变体离线预处理，也可以选择在渲染场景是按需创建。

3.3 着色器组件设计的好处：

  • 模块化、可拓展的代码框架：每一种 shading feature 都对应一个 component type，提供了清晰的边界，将参数定义和运算逻辑本地化。添加一种 shading feature，只需要添加一个新的 component type，作为已经存在的 component interface 的一种新的实现，而不需要在库中进行大量的修改。

  • 高效的参数传递：因为每个组件实例都封装了一个参数块，着色系统就可以在 GPU memory 上预申请并填入参数块，而不会在渲染时产生额外的 CPU-GPU 参数传递开销。着色器编译器需要保证由某一个 component type 创建的参数块，能够被所有使用该 component type 的着色器程序使用。

  • 简化着色器内核编译和选择：与使用预处理器宏来控制着色器代码生成不同，着色器组件设计通过设置着色器参数(如各种组件实例)实现。这种实现可以设计高效的 cache 查找机制(每一种着色器程序都是由 entry point 和各种组件实例组合而成)。相反，过去的着色系统将预处理器宏拼接而成的字符串传递给编译器，来作为着色器程序的标识符，而生成和比较这种复杂的字符串会带来很大的 CPU 开销。

4.1 Slang 设计约束与原则：

  • 支持轻松实现新的 shader feature
  • 语言功能与特点应符合开发者习惯
  • 与现有着色器语言的兼容性，在保留原语言可编译的基础上拓展其功能
  • 不预设新的 policy，使 Slang 能够支持各种设计模式
  • 及时并精确地报告错误

4.2 语言机制：

4.2.1 & 4.2.2 Generics、interface bounds
  • 在这个例子中，B 就是一种代表了未知 BxDF 的 Generic 类型参数，其中组件接口 IBxDF 是一种 interface bound，用于为该 Generic 类型提供边界，只有实现了该接口的组件类型能够通过编译。
  • 与 C++ 的模板不同，C++ 的模板在每次 specialization 的时候都会进行检查(调用该类型方法时，检查该类型是否包含该被调用的方法)，而在 Slang 语言中，因为只有实现了该接口的组件类型(必定拥有其被调用的方法)能够通过编译，这种显式的接口边界可以保证一致性的检查在着色器代码编译时就进行，而不需要等到使用该类型才报错。
  • 在上面的例子中，Slang 可以直接根据该 interface bound(IBxDF) 进行编译检查。
  • 泛型类型在特化着色器变体中会被替换成具体的类型，调用的方法也会是所替换的类型的方法，从而实现了静态特化，而非动态分发(与 C++ 中的多态不同)。

4.2.4 Retroactive Extensions
  • 在某些情况下，开发者可能想在框架中为已有的 feature 进行扩展，通过 extension 即可将新的方法加入，或是将该实现作为某个接口的实现(接口的要求由编写 extension 的开发者负责)
4.2.5 显式参数块

  • Slang 支持如上普通的结构体类型作为参数块，结构体中的成员即为参数。

  • 不同的图形API实现的参数块内存布局不同，因此 Slang 提供了抽象的内存块布局，在其标准库中表示为 ParameterBlock< T >，其内部的实现根据不同的平台也会有所不同。

  • 例如在 Direct3D 12 中， Slang 根据 T 的定义有两种映射方法：如果 T 中只包含 ordinary data fields，则可以直接将其映射到 ConstantBuffer< T >中; 如果 T 中包含 ordinary data fields 和 resource-typed field，则需要分开处理，ordinary data fields 部分直接映射到 ConstantBuffer< T >中， resource-typed field 部分作为全局参数。

  • 在所有平台中，在为所有 resource-typed 参数生成 layout annotations之前，Slang 不会试图将未使用的参数去除，也不会将他们从反射信息中去除。

  • 为了实现高效的参数通信，Slang 语言的定义要求所有编译器实现保证参数块的布局在所有使用该参数块的着色器变体中都是相同的。这种保证使着色系统能够安全地复用在给定类型后创建的参数块，并将其用于需要相同类型参数块的所有着色器变体。着色器组件设计模式依赖于这一保证，因为根据组件类型创建的组件实例必须与使用相同组件类型的任何着色器变体兼容。

4.2.6 全局 Generic 参数
  • 上面的两个 entry point 实际上是等价的，后者使用了全局泛型参数。
  • 尽管使用显式泛型拥有很多好处，但对于大型的 HLSL shader library 来说，一次性将着色器所有的全局参数都重构成每个 entry point 的显式函数参数并不现实，因此提供了全局泛型参数作为过渡，从而可以进行逐步重构。

4.3 编译器运行时API：

  • 开发者将 component types 和 entry points 定义在模块中，模块会在编译器前端进行类型检查并生成IR。
  • 编译器运行时会提供反射 API，用于创建参数块时查询参数布局
  • 编译器运行时还会提供编译 API，用于生成特化的着色器内核代码
  • Slang 提供的运行时 API 并非严格意义上的运行时，其支持离线处理资产时、加载时或渲染时调用相关 API
4.3.1 加载着色器模块
  • loadModule 函数用于加载着色器模块，会触发类型检查与中间代码生成，如果发生错误则会通过 errMsg 返回错误信息，否则返回包含类型与函数中间表示的模块。
  • 加载得到的模块保存了类型与函数的定义，可以用于后续查询参数布局、生成特化着色器内核等任务。
4.3.2 用于参数传递的反射API

  • 上面是用 C++ 实现的创建一个组件实例的代码框架：

    • 第一步调用 findType 函数得到该组件类型的 type symbol
    • 第二步调用 getTypeLayout 函数得到该组件类型的参数布局。本例子是面向 Direct3D 12，每个参数块包含一个 descriptor table 和一个 constant buffer，descriptor table 用于存储资源类参数，如 texture 等，constant buffer 用于存储基础类参数，如 int，float 等。资源类参数可以通过调用 getResourceUsage 函数得到，该函数返回 ResourceUsage 类型的对象，基础类参数可以通过调用 getConstantBufferSize 来获取所需的 constant buffer 大小。

  • 理论上可以通过 setDefaultLayoutRule 来人为规定参数的布局，但并不推荐这样做。

  • 在着色器组建设计中，为了实现高效的参数传递，各个特化的着色器变体是可以复用参数块的，这意味着参数的布局信息需要在任何着色器变体生成之前得到。而在 HLSL 语言中，只有在运行时才能够在一个已有的着色器变体中查询得到参数的布局信息，同时，HLSL 也不能够保证不同的着色器变体之间参数的布局信息保持一致，这种限制导致了参数块的高效复用无法实现。

4.3.3 特化着色器代码
  • 在生成特化的着色器变体时，首先调用 findEntryPoint 函数来获取 entry point 的 symbol
  • 随后根据所需的组件类型，调用 specializeEntryPoint 函数生成特化的着色器变体内核代码
  • 在着色器组件设计中，并不要求何时生成特化的着色器代码，可以选择在绘制物体前编译生成，也可以选择先离线生成所有可能会用到的着色器变体代码，并存储到 cache 中。
  • Slang 也支持生成特化的泛型类型，可以通过 specializeType 实现
  • 在生成着色器变体代码时，唯一需要做的就是将所有所需的中间代码连接起来，最终生成目标平台的代码。类型检查和中间代码生成只在未特化时进行一次，在生成着色器变体代码时，可以直接复用。
4.3.4 查找着色器变体
  • 为了避免在运行中编译着色器代码，大多数着色系统会离线生成所有的着色器变体代码并存入 cache 中，Slang 提供了一种动态 ID 的方法来查询对应变体，例如，一个动态 ID 用 16-bit 的整数表示，那么一个 128-bit 的整数足够用来表示包含多达 7 个泛型类型参数的 entry point，使用这个 128-bit 的整数在哈希表中查找比直接使用字符串比较更加高效。
    (entry point + 7 * type) * 16 = 128
  • 为了支持这种实现，Slang 提供了用于查找 entry points 和 types 的 API ( getTypeUID 和 getEntryPointUID )
4.3.5 总结
  • Slang 没有引入任何特殊的着色系统约束，例如什么时候生成着色器变体代码或如何传递着色器参数，这使得 Slang 成为一个能够被不同的着色系统应用的通用着色器编译服务。

5.1.5 生成和查找着色器变体
  • 原先的 Falcor 需要根据 #define 来查找着色器变体，但是每一帧每个物体的绘制都需要查找着色器变体，因此基于字符串的查找会带来大量的 CPU 开销。
  • 重构后的 Falcor 使用 variant key 进行查找，由 entry point 的动态 ID 和使用的组件类型组成 (参照4.3.4节)，其中动态 ID 是使用组件类型定义而非组件实例
